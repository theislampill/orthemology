# R3 counterexample ledger

**Date:** 2026-07-20. Eight attacks mandated by R3 §5.3, each dispositioned as *representable* (the R3 definitions express it without contradiction) or *excluded-by-explicit-constraint* (the definitions now say why it cannot arise). The R2 ledger's 16 patterns remain in force unchanged.

| # | Attack | Disposition |
|---|---|---|
| CE-R1 | **Objectively inapplicable axis vs unresolved axis.** A pediatric axis on an adult case vs an axis the actor hasn't examined. | **Representable, now distinguished:** inapplicability is the explicit `not-applicable` value *in the profile* (a fact under `A`); unresolvedness is openness *in `p̂`*. Conflating them was possible in the R2 factorized reading ("at most one value" was ambiguous between empty-because-inapplicable and empty-because-unknown); the R3 exactly-one-on-applicable-axes rule + declared null removes the ambiguity. |
| CE-R2 | **Empty complete profile.** A case instantiating nothing in the repertoire. | **Representable where declared:** admissible exactly where `A`'s constraints admit it; an analysis whose repertoire exhausts the possibilities excludes it *by its own constraint*, not by framework fiat. No contradiction with `O*(m; A) ∈ Π_A`. |
| CE-R3 | **Overlapping axes.** One ortheme legitimately sitting on two axes. | **Representable with declaration:** partition-or-overlap is a declaration of `A`; overlap without reconciliation constraints is a well-formedness defect of the analysis, caught at declaration, not silently tolerated at placement time. |
| CE-R4 | **Hierarchical profile not naturally factorized.** Nested severity-within-cause structure. | **Representable in the general form:** `Π_A ⊆ 𝒫(O_A)` with hierarchical admissibility constraints; factorization is one representation family (Definition 10, R3), so nothing forces flattening. |
| CE-R5 | **Conflicting governance requirement rules.** Two rules disagreeing whether V6 is required. | **Excluded by structure within an instance:** the shipped table has exactly one rule per verdict (checked by `derive_reqpath.py`: every CorePath verdict has exactly one rule), so intra-instance conflict cannot arise; conflicting *regimes* are a configuration replacement, and loading two tables at once is a load-time defect. |
| CE-R6 | **Required verdict omitted by an otherwise valid record.** Token-bearing episode declaring a path without `GOV_TOKEN_ADEQUATE`. | **Detected:** fixture RP-5; derivation recomputes the required set from shape, and declared ≠ derived is a failure, never a tolerated subset. |
| CE-R7 | **Analysis version change that changes axis applicability.** `A v1` has a universally applicable axis; `A v2` makes it conditional. | **Representable via version transport (manuscript §2.6):** profiles and placements are indexed to `(A, version)`; nothing verified under v1 transports to v2 without a lineage argument; the `not-applicable` value is version-relative like every other profile fact. |
| CE-R8 | **Fusion analysis with nonunique mapping.** Two admissible embeddings of component profiles into a fusion analysis. | **Acknowledged open parameter (risk-accepted):** fusion non-uniqueness was already on the R2 limitation ledger and remains; the R3 addition is only the typing discipline that any chosen fusion mapping must be declared with the fusion analysis so that cross-analysis comparisons stay well-typed. Not silently closed. |

**Summary:** 6 representable (2 of them by newly explicit distinctions), 1 excluded by structure, 1 risk-accepted open parameter — matching the honest lane verdict in [`R3-FORMAL-AUDIT.md`](R3-FORMAL-AUDIT.md).
